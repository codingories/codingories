  1. Stream - 流
  • 释意
  • stream 是水流，但默认没有水
  • stream.write 可以让水流中有水（数据）
  • 每次写的小数据叫 chunk(块)
  • 产生数据的一段叫做 source(源头)
  • 得到数据的一段叫做 sink(水池)
  1. 第二个例子
  • 分析
    • 用任务管理器查看 Node.js 内存占用，大概 130 MB
  1. 第三个例子
  • 用 Stream 改写二个例子
  • 分析
    • 查看 Node.js 内存占用，基本不会高于 30 Mb
    • 文件 stream 和 response stream 通过管道相连
  1. 管道
  • 释意
    • 两个流可以用一个管道相连
    • stream1 的末尾连接上 stream2 的开端
    • 只要 stream1 有数据，就会留到 stream2
  • 常用代码
    • stream1.pipe(stream)
  • 链式操作
  1. 管道续
  • 管道可以通过事件实现
  1. 总结
  • 流的好处，可以使内存降的很低
  • 两个流可以连起来，用管道
  1. Stream 对象的原型链条
  • 如何研究一个对象
    • node –inspect-brk 4.js,打断点
      • s = fs.createReadStream(path)
      • 那么它的对象层级为
      • 自身属性(由 fs.ReadStream 构造)
      • 原型: stream.Readable.prototype
      • 二级原型: stream.Stream.prototype;
      • 三级原型: events.EventEmitter.prototype
      • 四级原型: Object.prototype
    • Stream 对象都继承了 EventEmitter
  • 常用支持的事件和方法
    • 可读的 stream 的重要事件 data, end
    • 可写流，drain 流干,drain 表示之前拥堵的车道不再拥堵了，非常湍急的水流已经干涸了
    • *** drain 面试常考,但是用的不多，除非碰到非常复杂的业务
    • 可写流 finish，整个写完了
  1. Stream 的分类共四种
  • 可读
  • 可写
  • Duplex 可读可写（双向）
  • Transform 可读可写（变化）,scss,webpack 打包都用到 transform
  1. readable 和 writable 的特点

  • Readable Stream

    • 静止态 paused 和流动态 flowing
    • 默认处于 pause 态
    • 添加 data 事件监听，它就变为 flowing 态
    • 删掉 data 事件监听，它就变为 paused 态
    • pause()可以将它变为 paused
    • resume()可以将它变为 flowing

  • Writable Stream

    • drain 流干了事件
      • 表示可以加点水了
      • 我们调用 stream.write(chunk)的时候，可能会 false
      • false 的意思是你写的太快了，数据积压了
      • 这个时候我们就不能再 write 了，要监听 drain
      • 等 drain 事件出发了，我们才能继续 write
    • finish 事件
      • 调用 stream.end()之后，而且
      • 缓冲区数据已经传给底层系统之后
      • 触发 finish 事件

  1. 关系型数据库的范式，标准
  2. 第一范式
  • 定义
    • 字段不可再分
  • 举例
    • 我们要存储体验者的双眼视力
    • 那么应该存为左眼视力和右眼视力两个字段
    • 即 user 表里应该有 left_eye 和 right_eye
    • 而不能把他们存在一个字段
  • 第一范式的缺点
    • 数据冗余，创建的时候造一些假数据，删除导致字段消失
    • 结论：第一范式不够强
  1. 第二范式
  • 定义（不标准）
    • 在 1NF 的基础上，要有键(键可由多个字段组合)
    • 所有字段分别完全依赖于键
    • 如果键是多个字段组合，则不允许部分依赖于该键
  • 依赖关系
    • 给出键，就能唯一确定字段的值
    • 如给出学号，就能唯一确定姓名，反之则不行
    • 则称姓名依赖于学号
  1. 第三范式
  • 定义（不标准）
    • 一个表里不能有两侧依赖
    • 给出学号，就能确定系名:系名依赖于学号
    • 给出系名，就能确定系主任:系主任依赖于系名
    • 所以，系主任间接依赖于学号
  • 解决办法
    • 把系名和系主任单独建表
  1. 总结
  • 第一范式
    • 属性不可分割
  • 第二范式
    • 字段完全依赖于键
  • 第三范式
    • 字段没有间接依赖于键
  • BC 范式
    • 键中的属性也不存在间接依赖
  1. 数据库设计经验
  • 高内聚
    • 把相关的字段放到一起，不相关的分开建表
    • 如果两个字段能够单独建表，那就单独建表
  • 低耦合
    • 如果两个表之间有弱关系
    • 一对一可放在一个表，也可两个表+外键
    • 一对多一般用外键
    • 多对多一般建中间表

      • GET 请求: curl -v url
      • POST 请求: curl -v -d “name=ories” url
      • 设置请求头: -H ‘Content-Type: application/json’
      • 设置动词: -X PUT
      • JSON 请求: curl -d ‘{“name”:”bob”}’ -H ‘Content-Type:application/json’ url
      • 后面会用到 curl 来构造请求

  3. request 对象

    1. server 是个什么东西

      1. http.Server 类的实例
      2. 根据文档知道 http.createServer 的返回值的类型
      3. 所以 server 是 http.Server 的实例
      4. 因此 server 拥有几个事件和方法
      5. 其中也就 request 事件和 listen()方法目前用的上

    2. 继承 net.Server 类

      1. 根据文档知道 http.Server 继承了 net.Server
      2. 因此 server 有用了几个事件和方法
      3. 其中也就 error 事件和 address()方法目前能用上,address 看监听的对象是什么

    3. 看文档的方法，看有哪些事件，属性，方法

      • 有用的事件, request 有人请求了怎么办, listen 开始监听
      • curl + -v 可以打印出请求的内容
      • 处理 post 请求无法发现，请求体

  4. 如何得到请求消息体

  5. 用 Node.js 获取请求内容

    • request.method 获取请求动词
    • request.url 获取请求路径（含查询参数）
    • request.header 获取请求头
    • get 请求一般没有消息体/请求体

  6. post 请求

    • curl -v -d “name=ories” http://localhost:8888

    • request.on(‘data’,fn)获取消息体

    • request.on(‘end’,fn)拼接消息体

  7. (request, response)是啥

    1. 找类

      • 根据文档， request 是 http.IncomingMessage 的实例
      • 根据文档, response 是 http.ServerResponse 的实例

    2. Request

      • 拥有 headers,method,url 等属性
      • 从 stream.Readable 类继承了 data/end/error 事件
      • 为什么不能直接拿到请求的消息体，跟 TCP 有关

    3. Response

      • 拥有 getHeader/setHeader/end/write 等方法

      • 拥有 statusCode 属性，可读可写

      • 继承了 Stream,目前用不上
