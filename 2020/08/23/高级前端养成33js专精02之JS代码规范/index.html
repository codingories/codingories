  1. 质量意识
  • 一些错误但大行其道的观点
    • 我们没有时间写单元测试
    • 应该给每一行代码加注释
    • 代码中的空格，括号非常重要
    • 等我们有时间了再重构
    • 只要坑挖得足够多的，就可以筑起护城河
    • 测试是测试人员的事情，跟开发没关系
  • 错再哪
    • 没时间写测试，有时间改 bug
    • 正确地写注释，注释 why 而不是注释 how
    • 空格，括号远没有那么重要
    • 重构是要每天进行的
    • 只要坑挖得足够深，头发就会掉得更快
    • 应该能独立保证自己的代码质量，不要完全依赖测试人员
  1. 现状
  • 人员水平
    • 大部分程序对质量的提升方法是堆时间(加班)
    • 正确的办法是自动化，监控，预警
    • 大部分前端对代码质量还停留在 Lint 工具
    • 正确的做法是静态分析，单元测试，不断重构
  • 需要做的
    • 不要被错误的观点同化
    • 相信一切能用机器做的事情，都应该交给机器做
    • 一旦有实践自动化，测试的可能，就勇敢地上

  1. JS 语言特性-语言提供给我们的各种功能

    • 道格拉斯的观点
      • 如果一个特性有时好用有时危险，还有一个更好的选项，那么我们总是应该使用那个更好的
      • 公司雇我们不是为了让我们使用语言的每个特性，而是让我们写出可以用的代码，并且避免错误

  2. 程序员的共识

  • 1960 年代，我们花了一代人的时间去认同
    • 高级语言是好的(反方: 机器语言效率高)
  • 1970 年代，我们花了一代人的时间去认同
    • goto 是不好的，支持结构化编程
  • 1980 年代，我们花了一代人的时间去认同
    • 对象是好的，发展面向对象
  • 1990 年代，我们花了两代人的时间去认同
    • lambda 表达式是好的(Java,C++,PHP,JS 加强支持)
  1. 存在即合理
  • 错误，有些就是不合理，比如 var 和==
  1. 代码越短越好
  • 错误， 无歧义优于简洁
  1. JS 代码中的特性取舍
  • 用哪些?不用哪些？
  • 能完成特定功能无歧义，无副作用
    • 这样的特性，优先使用
  1. 不要用的 JS 特性
  • 绝对不要用
    • 全局变量
    • var 声明
    • ==运算符
    • 包装类型 String/Boolean
  • 有些人不用
    • class/new/this
    • 比如 React Hooks 那帮人
  • 用的人不多，尽量少用
    • symbol,generator,iterator
    • 反射
  1. 推荐使用的 JS 特性

  • 一定要用

    • === 全等
    • …三个点运算符
  • 模块 import export
  • let 和 const,优先使用 const
  • 析构赋值

  • Promise/await

  • 可以使用

    • class
    • getter/setter
  • WeakMap,代替 symbol
  • Proxy,Vue3 核心取代 defineProperty
  • 存在问题但可用
    • 箭头函数
    • 数组 API+Polyfill,参考资料
  1. 命名
  • 坏命名示例
  • 首要规则，命名应该让人秒懂
  • 命名应该符合英语习惯
    • 普通变量用名词
    • 复数要加 s 或者对应复数形式
    • 布尔用 isXXX 或 canXXX 或形容词
    • 静止用 showDialog 表示是否展示 dialog
  • 普通函数用动词开头
  • 钩子函数用介词加动词开头

  • 容易混淆的对象加前缀，如$div

  • 命名不准用缩写

    • 如 cnt/idx/cur/prev/anal
    • 某些特定的缩写可以用，比如 for 循环里面的 ijk
    • 行业通用缩写当作

  • 注释分类

    • 对代码进行翻译的注释
    • 对代码进行总结的注释
    • 对 bug 进行分析的注释
    • 对参考来源的注释
    • 对潜在问题的警告的注释
    • 发泄情绪的注释
  1. 推荐工具
  • lint 工具
    • eslint
    • Standard JS
    • 不推荐 jslint/jscs/jshint
  • 推荐配置(也不是那么推荐)
    • eslint-config-Airbnb
  • 建议
    • 不要那么在意空格，括号
    • 请关注命名，代码解构，单元测试
